# Weekly Report Writing

这个 skill 用于帮助用户整理、撰写、润色或评审职场周报，尤其适用于项目周报、实习周报、研发周报、实验进展周报、导师/leader 汇报材料。

核心目标不是“把事情写满”，而是让读者快速判断：

- 你实际推进了什么；
- 你解决了什么问题；
- 这些动作为什么重要；
- 已经得到哪些阶段性结论；
- 新结果暴露了什么现象；
- 下一周期必须调整和完成什么。

## 触发场景

当用户提出以下需求时，应调用本 skill：

- “帮我写周报”
- “整理一下本周工作”
- “把这些工作写成周报”
- “帮我润色周报”
- “检查我的周报有没有问题”
- “根据实验结果写周报”
- “给 mentor / leader / 组会看的周报”
- “项目进展汇报”

## Mentor 原话

以下内容应作为周报整理的核心原则，尽量保留原意和措辞：

> 周报整理书写是职场很重要的技能，是前期需要花些心思去储备的技能，大概要有这几个东西
>
> 进展：写你实际做了什么，解决了什么问题，为什么要做
>
> 结论：前几周你做的事情得到的哪些阶段性结论，如实验效果、指标等
>
> 分析：最新的结果呈现出了新的现象，针对做的思考和分析
>
> TODO：针对分析的结果，哪些动作要微调，下一个工作周期必须完成的事项

## 图像参考原则

用户上传的图像强调了“项目周报怎么写”的几个关键点，写作时应吸收这些原则：

### 先判断周报给谁看

周报有多种：

- 给自己看的；
- 给别人看的；
- 组会上的周报，通常是给别人看的；
- 读者可能是小组其它同事，也可能是 leader。

给谁看决定了应该怎么写。面向他人的周报，要减少只有自己懂的过程细节，增加结果、结论、影响和下一步。

### 描述型表达要带量化信息

主要体现两点：

- 做了什么；
- 做得怎么样。

在写“做得怎么样”时，应尽量给出明确、具体、可量化的信息，而不是只写主观感受。

可量化信息包括：

- 阶段性进度；
- 数值变化；
- 收益；
- 指标变化；
- 实验效果；
- 消耗时间；
- 当前剩余风险或缺口。

主观感受也是一种“不易得”的量化，应尽量转化为事实、现象、对比或判断依据。

### 多用表格和图表

如果一周内有多个同类事项、多个实验、多个方案或多个 bad case，应优先考虑用表格呈现。

适合用表格的场景包括：

- 多个实验结果对比；
- bad case / good case 对比；
- 不同方案优劣对比；
- 本周事项的状态汇总；
- 指标变化前后对比；
- TODO 优先级排序。

### 万事均可量化

无法量化的内容，往往说明还没有想清楚这个项目或需求的目标是什么。

写周报时，如果一句话无法量化，需要追问：

- 这件事的目标是什么？
- 解决了什么问题？
- 进展到什么程度？
- 产生了什么收益？
- 还差什么才能闭环？
- 为什么这件事值得占用本周时间？

### 堆砌型工作要解释收益

对于一些零碎、繁杂、难体现价值的事项，可以打包说明。

但如果这些事情占用了不少工作时间，却没有带来明确收益，就需要说明：

- 为什么必须做；
- 做完后减少了什么风险；
- 为后续工作创造了什么条件；
- 如果不做会造成什么阻塞；
- 下一步如何让这些投入转化为结果。

## 标准周报结构

默认使用以下结构组织周报：

## 进展

写清楚本周实际做了什么，解决了什么问题，为什么要做。

每条进展建议包含：

- 事项；
- 背景或目标；
- 具体动作；
- 当前状态；
- 可量化结果。

避免只写：

- “调研了 xx”
- “处理了 xx”
- “跟进了 xx”
- “优化了 xx”

更好的写法：

- “完成 xx 模块接口联调，解决 yy 场景下返回为空的问题，目前主流程已跑通，剩余 2 个边界 case 待验证。”
- “对 xx 和 yy 方案做性能对比，在 3 组数据上验证后，xx 方案耗时降低约 18%，但在长文本场景稳定性仍不足。”

## 结论

写前几周或本周持续工作的阶段性结论，如实验效果、指标变化、方案判断。

结论应尽量是判断句，而不是过程描述。

优先写：

- 哪个方案更好；
- 哪个假设被验证；
- 哪个问题被证伪；
- 当前指标是否达到预期；
- 阶段性结果对后续方向意味着什么。

避免只写：

- “做了实验”
- “看了数据”
- “效果还可以”
- “有一些提升”

更好的写法：

- “当前实验表明，xx 策略对长尾 query 的召回提升更明显，整体指标从 30% 提升到 36%，但高频 query 收益不稳定，下一步应拆分场景继续验证。”

## 分析

写最新结果呈现出的新现象，以及针对这些现象的思考。

分析不是复述数据，而是解释：

- 为什么会出现这个结果；
- 和预期是否一致；
- 哪些因素可能影响了结果；
- 当前现象说明了什么；
- 是否需要调整原方案；
- 还有哪些不确定性。

可使用以下句式：

- “这个结果和预期不一致，可能原因是……”
- “从 bad case 看，问题主要集中在……”
- “目前收益主要来自……，但在……场景下仍存在明显缺口。”
- “这说明当前方案解决了……，但还没有解决……”

## TODO

TODO 必须来自分析，而不是机械罗列任务。

每个 TODO 应写清楚：

- 要做什么；
- 为什么要做；
- 优先级；
- 完成标准；
- 是否必须在下一个工作周期完成。

优先写“下一周期必须完成的事项”，避免把所有想做的事都堆进去。

更好的 TODO 写法：

- “补充 xx 场景 50 条 bad case 标注，验证当前策略是否只在短文本场景有效；完成标准是形成按问题类型拆分的 case 表。”
- “将 xx 实验从离线样本扩展到线上近 7 天数据，确认 36% 的提升是否可复现；完成标准是产出对比表和下一步方案判断。”

## 输出风格

写周报时应遵循以下风格：

- 简洁、具体、面向读者；
- 用事实和数字替代形容词；
- 用结论句替代流水账；
- 对不确定性明确标注，不假装已经闭环；
- 面向 leader 或组会时，突出结果、风险和下一步；
- 面向自己复盘时，可增加过程细节和经验沉淀。

## 质量检查

生成或修改周报后，应按以下标准自检：

- 是否说明了“给谁看”；
- 是否写清楚“做了什么”；
- 是否写清楚“做得怎么样”；
- 是否解释了“为什么要做”；
- 是否包含阶段性结论；
- 是否有实验效果、指标、进度或其它量化信息；
- 是否对新现象做了分析，而不是只贴结果；
- TODO 是否来自分析；
- TODO 是否有明确完成标准；
- 是否存在“忙了很多但价值不清楚”的事项；
- 是否能让别人了解你的工作和进度；
- 是否能展示你的能力和成果。

## 常见问题

### 缺少结果

Bad case：

> 上线 xx 平台。

Good case：

> 上线 xx 平台，实现 xx 功能，目前 xx 人在用，节省了 yy 流程中的人工检查时间。

### 缺少结论

Bad case：

> 对 xx 和 xx 做了性能对比。

Good case：

> 对 xx 和 xx 做了性能对比。在 xx 场景 A 更好，在 yy 场景 B 更好，因此下一阶段优先优化 A 在 yy 场景下的表现。

### 缺少数据支持

Bad case：

> 数据标注中；优化转场策略；VOS 提升明显。

Good case：

> 数据标注进度 50%，预计还需要 3 天；优化转场策略后，VOS 从 30% 提升到 36%，但在 xx 类型样本上仍有明显波动。

## 推荐模板

```markdown
# 本周周报

## 进展

- [事项 1]：本周完成了……，解决了……问题。做这件事的原因是……。当前进度/结果是……。
- [事项 2]：……

## 结论

- [结论 1]：基于……实验/数据/观察，目前可以判断……。关键依据是……。
- [结论 2]：……

## 分析

- [现象 1]：最新结果显示……，这个现象说明……。可能原因是……，仍需验证的是……。
- [现象 2]：……

## TODO

- [必须完成] ……；原因是……；完成标准是……。
- [需要推进] ……；原因是……；完成标准是……。
```

## 使用方式

当用户给出零散工作记录时，应主动将其重组为：

- 进展；
- 结论；
- 分析；
- TODO。

如果信息不足，不要编造数字或结论，应直接指出缺口，并用问题引导用户补充：

- 这个事项的目标是什么？
- 当前做到什么程度？
- 有没有实验指标或结果？
- 相比上周有什么变化？
- 这个结果说明了什么？
- 下周必须完成什么才算推进？

如果用户只要求润色，应尽量保留其原有事实，不擅自新增结果；可以把模糊表达改为更清晰的结构，但必须标注需要用户补充的数据。
